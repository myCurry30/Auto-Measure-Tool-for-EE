# Skill: 仪器控制软件开发

## 触发条件

当用户要求开发、设计或讨论以下内容时触发：
- 科学仪器的控制软件/程序（SMU、锁相放大器、温控器、位移台、光谱仪等）
- 通过GPIB/USB/LAN/Serial连接仪器的Python程序
- 需要实时反馈控制的实验自动化（电迁移、PID控制、自适应测量等）
- 仪器软件的UI/UX设计
- PyVISA、SCPI、TSP相关的仪器通信

## 详细参考

完整方法论见 `instrument_software_playbook.md`（同目录）。以下为核心要点速查。

## 架构：三层分离

```
UI层（Web/Desktop/CLI）
    ↕ Queue解耦
实验逻辑层（algorithm）
    ↕ 方法调用
仪器驱动层（driver）— 独立可复用包
```

- **驱动层**：只封装仪器原子操作（设置电压、读电流、查状态），不含实验判断。以仪器型号命名为独立包，可跨项目复用。
- **实验逻辑层**：调用驱动原子操作组合实现实验流程。在独立线程运行。每个测量点后执行安全检查链。
- **UI层**：通过Queue订阅数据，与实验逻辑完全解耦。UI崩溃不影响实验运行。

## 安全检查链（每个测量点后执行）

1. Compliance/限值状态
2. 开路检测
3. 短路检测
4. 绝对安全上限（硬编码）
5. 通信超时/错误
6. 用户中止指令

必须实现`emergency_off()`：归零→关输出，作为最快安全路径。

## UX核心原则

**设计方向**：参考现代Web产品风格（Linear/Notion/Vercel），不参考仪器行业UI。数据图表是主角。

### 布局

侧边栏(280-320px) + 内容区二栏布局。顶栏放仪器标识和全局状态。

```
侧边栏从上到下：Preset → 常用参数 → Advanced(折叠) → 派生信息 → 控制按钮 → 实时状态
内容区从上到下：图表（占最大面积） → 事件日志 → 后处理分析
```

- 图表区占比 > 控制面板，是界面视觉中心
- 每张图至少200-250px高，宁可滚动不要压缩到看不清
- 两图并排用等宽两列；最重要的图全宽
- 顶栏高度克制(48-56px)，不浪费纵向空间

### 配色

四色语义系统，全局一致：

| 语义 | 用途 |
|------|------|
| **蓝** | 主操作、选中、信息 |
| **绿** | 就绪、成功、连接 |
| **琥珀** | 运行中、警告、需注意 |
| **红** | 错误、危险、紧急（保留给不可逆/危险操作） |

每色三变体：正常色(文字/线条) + 淡底(状态徽章背景) + 边框。

背景用2-3级灰度创造层次（页面底 → 卡片 → 输入框）。不用纯黑#000或纯白#fff做大面积背景。文字三级：主要 / 次要 / 弱化。

图表配色按数据语义：时间演化用单色浅→深或蓝→红渐变；分类对比用不同色相+线型冗余；极性用蓝(+)/红(-)。网格线极淡，不抢数据焦点。

颜色不做唯一编码——始终配合文字/图标/线型做冗余（色盲友好）。

### 字体

双字体系统：等宽(IBM Plex Mono/JetBrains Mono)用于所有数值和数据；无衬线(Outfit/Inter/DM Sans)用于UI文字。数值区域预留固定宽度+右对齐，变化时不抖动。

### 减少输入 & 防止错误

| 机制 | 作用 |
|------|------|
| **Preset系统** | 选preset自动填参数，支持保存/导入JSON |
| **参数分级** | 常用3-5个始终可见，高级参数折叠 |
| **三级验证** | ❌错误（阻止启动）⚠️警告（需确认）ℹ️提示（tooltip） |
| **派生信息** | 自动计算步数/耗时/功率等，帮用户判断合理性 |
| **启动确认面板** | 参数总览 + 连通性自检 + 元数据必填，三合一 |

### 数值显示

- 永远用工程记数（SI前缀）：4.27 kΩ，不用 4270 Ω 或 4.27e3
- 3-4位有效数字，等宽字体右对齐
- 输入框接受多格式："100m" / "1e-1" / "0.1" / "5%"
- 图表Y轴单位随数据范围自适应

### 运行中保障

- 参数分为只读（运行时不可改）和可修改两类
- 关键事件声音/标题提醒
- UI断连不停止实验，重连恢复状态
- 操作日志记录所有用户动作

## 数据安全

- 每点立刻flush，不攒buffer
- 文件名含时间戳，不覆盖
- 参数JSON与数据文件绑定保存
- 元数据（样品ID、描述）随数据存储
- 启动时检测仪器遗留输出状态

## AI协作工作流

```
读文献/手册 → 讨论确认需求 → 写Spec(接口+伪代码+数据格式)
→ CLAUDE.md指向Spec+手册 → 按模块顺序开发 → 逐模块测试
→ 前端：设计说明+HTML原型 → 实现
```

开发顺序：驱动 → 配置 → 安全 → 算法(先Mock) → 存储 → 分析 → UI → 整合

**关键**：驱动层必须在真实仪器上人工测试。通信协议（SCPI/TSP/私有）写驱动前必须确认并查手册。
