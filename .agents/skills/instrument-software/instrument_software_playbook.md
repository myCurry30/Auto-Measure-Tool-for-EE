# 仪器控制软件开发 Playbook

本文档总结了基于Python开发科学仪器控制软件的完整方法论，涵盖架构设计、UX原则、防错机制和AI协作工作流。源自iFBEM（电迁移纳米间隙制造）控制软件的开发经验，但原则适用于所有仪器控制软件——SMU、锁相放大器、温控器、位移台、光谱仪等。

---

## 1. 架构设计

### 1.1 三层分离原则

所有仪器控制软件应分为三个严格解耦的层次：

```
┌─────────────────────────────┐
│   UI / 可视化层              │  ← 显示数据、接收用户输入
│   (Web / Desktop / CLI)     │
├──────────── Queue ──────────┤  ← 数据队列解耦
│   实验逻辑层                 │  ← 组合原子操作实现实验流程
│   (algorithm / procedure)   │
├─────────────────────────────┤
│   仪器驱动层                 │  ← 纯仪器原子操作
│   (driver)                  │
└─────────────────────────────┘
```

**仪器驱动层**只封装仪器本身能执行的原子操作，不包含任何实验判断逻辑。判断标准：如果一个方法的实现只需要发送/接收仪器命令，它属于驱动层；如果需要根据测量结果做决策，它属于实验逻辑层。

驱动层的设计要点：
- 每个方法对应一个仪器动作（设置电压、读取电流、查询状态）
- 不组合多个仪器操作为一个方法（如"扫描I-V曲线"不属于驱动层）
- 双通道/多通道仪器通过channel参数支持，不写死通道
- 支持Context Manager（`with`语句），退出时保证输出关闭、连接释放
- 有`emergency_off()`方法，做最快速的安全关闭（先归零再关输出）
- 不在驱动层做数据处理（不算电阻、不做平均、不做判断）

实验逻辑层的设计要点：
- 通过调用驱动层的原子操作组合实现实验流程
- 在独立线程/进程中运行，不阻塞UI
- 通过Queue与UI层通信（data_queue推数据、control_queue收指令）
- 每个测量点之后执行安全检查链
- 支持pause/resume/abort控制

UI层的设计要点：
- 与实验逻辑完全解耦，通过队列或事件订阅数据
- 可以是Web前端、桌面GUI、或纯CLI——实验逻辑不应关心UI的存在形式
- 即使UI崩溃/断连，实验逻辑应能独立运行或安全停止

### 1.2 可复用驱动的设计

驱动层应作为独立包（独立目录、独立`__init__.py`），不依赖实验代码中的任何内容。目录结构：

```
keithley2636b/        # 以仪器型号命名
├── __init__.py       # 导出主类
├── driver.py         # 驱动主类
└── constants.py      # 仪器常量的Python映射
```

这样做的好处是：同一个驱动可以被I-V扫描、电迁移、恒压应力、脉冲测量等完全不同的实验脚本复用。

### 1.3 通信协议注意事项

- 确认仪器使用的协议（SCPI / TSP / 厂商私有），不要想当然
- 写驱动前务必获取仪器编程手册，不要凭记忆猜命令语法
- 用PyVISA做VISA通信，支持GPIB/USB/LAN/Serial多种接口
- 每次`query()`后检查返回值格式，加入重试逻辑
- 设置合理的通信超时（通常3-10秒）

### 1.4 实时反馈控制实验的模式

适用于：电迁移FBEM、PID温控、恒电流电沉积、自动对焦、自适应采样等需要"测量→判断→响应"闭环的实验。

通用控制流：

```
初始化仪器和参数
测量初始基线

while 未达到终止条件 and 未被用户中止:
    设置激励（电压/电流/位置/温度）
    等待（如需settlement time）
    测量响应
    
    执行安全检查链:
        检查compliance/限值
        检查开路/短路/异常
        检查绝对安全上限
        检查通信状态
        检查用户控制指令
    
    推送数据到队列
    
    执行反馈判据:
        如果触发 → 执行响应动作（归零/调整/记录）
    
    执行终止判据:
        如果满足 → 退出循环

清理：安全关闭仪器输出
推送最终状态
```

反馈判据的设计模式：
- **单阈值**：简单但可能对噪声敏感，适合信噪比高的场景
- **双阈值**（推荐）：硬阈值立刻触发（捕捉突变），软阈值需连续N点确认（过滤噪声）
- **自适应阈值**：根据实验进程动态调整，初期宽松、后期敏感

安全检查链是一个有序的检查列表，每个测量点之后立刻执行，任何一项触发都应优先于反馈判据处理：
1. 仪器compliance/限值状态
2. 开路检测（电流低于噪声底限）
3. 短路检测（电阻异常低）
4. 绝对安全上限（激励值超限）
5. 通信超时/错误
6. 用户中止指令

### 1.5 紧急停止

紧急停止必须是系统中响应最快的路径。两种实现方案：

**方案A：通过控制队列。** UI发abort指令到control_queue，测量循环每个点检查。优点：线程安全。缺点：最大延迟等于一个测量周期。适合测量周期短（<100ms）的场景。

**方案B：threading.Event + 直接调用。** UI设置一个Event标志，同时直接跨线程调用`smu.emergency_off()`。优点：近乎即时响应。缺点：跨线程访问仪器对象，需要确保驱动层的emergency_off是线程安全的（通常是，因为它只发一条命令）。适合测量周期长或样品敏感的场景。

建议：先用方案A实现，如果实际测试发现响应太慢再切换到方案B。

## 2. 仪器软件UX设计原则

这是本文档最重要的部分。仪器软件的用户是科研工作者——他们关心的是实验数据和结果，不是软件本身。好的仪器软件应该让用户尽量少操作、尽量不犯错、尽量不丢数据。

### 2.1 设计方向

**不要参考仪器行业的UI设计语言。** LabVIEW、Keysight、NI、R&S等仪器公司的软件界面普遍过时且信息密度过高。应该参考现代Web产品（Linear、Notion、Vercel Dashboard等）的视觉风格：干净、克制、留白充足、信息层次清晰。

**数据可视化是主角。** 用户80%的时间在看图表，UI控件应该安静地退到旁边。图表区域占比应大于控制面板。

**浅色或深色主题取决于使用环境。** 光线充足的实验室用浅色主题；暗室/光学实验用深色。最好都支持，至少支持实际使用环境的那个。

**专业但不丑。** 用等宽字体显示数值（JetBrains Mono, IBM Plex Mono, SF Mono等），用无衬线字体显示UI文字。配色克制，强调色用于状态区分（蓝/绿/琥珀/红），不做装饰性渐变或动效。

### 2.2 布局

#### 通用布局模式：侧边栏 + 内容区

绝大多数仪器软件都是"设参数 + 看数据"的模式，适合左侧边栏放控制、右侧主区域放图表的二栏布局：

```
┌──── 顶栏 ───────────────────────────────────────────────┐
│  软件名称 · 仪器标识                    [状态指示]       │
├── 侧边栏 ────────┬── 内容区 ───────────────────────────┤
│                  │                                      │
│  Preset 选择器    │  图表区（占页面最大面积）             │
│                  │                                      │
│  常用参数         │  ┌─────────┐ ┌─────────┐            │
│                  │  │ 图表 1  │ │ 图表 2  │            │
│  [Advanced ▸]    │  └─────────┘ └─────────┘            │
│                  │  ┌───────────────────────┐            │
│  派生信息         │  │ 图表 3（全宽）        │            │
│                  │  └───────────────────────┘            │
│  [▶][⏸][⏹]       │                                      │
│                  │  事件日志                              │
│  实时状态数值      │                                      │
│                  │  后处理/分析区                         │
└──────────────────┴──────────────────────────────────────┘
```

布局原则：

**顶栏（始终可见，固定在顶部）：**
- 左侧：软件名称 + 仪器型号/地址（让用户确认连的是哪台仪器，多台仪器时避免混淆）
- 右侧：全局状态指示（Idle / Running / Error），这是用户最频繁扫一眼的信息，必须在视野最容易到达的位置
- 高度克制（48-56px），不浪费纵向空间

**侧边栏（固定宽度，可滚动）：**
- 宽度 280-320px，不要太宽抢图表空间
- 内容从上到下的优先级顺序：Preset选择器 → 常用参数 → 高级参数（折叠）→ 派生信息 → 控制按钮 → 实时状态数值
- 参数用卡片分组，每组有标题和视觉标记（彩色圆点或图标），快速区分不同类别
- 控制按钮（Start/Pause/Abort）放在参数之后、状态之前——逻辑上是"设好参数→执行→看状态"的自然流程

**内容区（填满剩余空间，可滚动）：**
- 图表占据最大面积，是整个界面的视觉中心
- 多张图表时考虑信息优先级：最重要的图大且靠上，辅助图可以小一些或排在下方
- 两张图并排时用等宽两列；有一张特别重要时让它全宽
- 事件日志放在图表下方，固定最大高度，超出可滚动——它是辅助信息，不应挤占图表空间
- 后处理/分析区（如曲线拟合）放在最底部，只在实验结束后才需要关注

#### 图表的布局策略

根据图表的数量和重要性选择布局：

**1张核心图**：全宽显示，占内容区上半部分。典型场景：简单的实时信号监控。

**2-3张同等重要的图**：两列并排，或一行两列 + 一行全宽。典型场景：I-V特性 + 电阻演化。

**1张核心 + 多张辅助**：核心图全宽靠上，辅助图小一些排在下方。典型场景：主信号 + 多个诊断量。

**仪表盘模式（4张以上）**：2×2或3列网格，每张图高度适当缩小。典型场景：多通道监控。

原则：宁可让用户滚动页面看更多图，也不要把所有图压缩到一屏里变得看不清。每张图都应该有足够的高度（至少200-250px）让数据可辨认。

#### 状态信息的放置

实时状态数值（当前测量值、周期数、运行时间等）有两种放法：

**方案A：放在侧边栏底部。** 优点是紧凑，不占图表空间。缺点是侧边栏可能需要滚动才能看到。适合状态量少（3-5个）的场景。

**方案B：放在顶栏里或顶栏下方的独立条带。** 优点是始终可见。缺点是占用纵向空间。适合状态量是用户最关心的信息、需要随时一眼看到的场景。

通常推荐方案A，把最关键的一条状态信息（如当前状态+周期数）放到顶栏的状态徽章里，其余放侧边栏。

### 2.3 配色系统

#### 原则：语义化配色

颜色在仪器软件里不是装饰，是**信息编码**。每种颜色必须有明确的语义含义，并且全局一致。

#### 四色状态系统

整个软件只需要四种语义色，加上中性灰：

| 语义 | 用途 | 浅色主题推荐 | 深色主题推荐 |
|------|------|-------------|-------------|
| **蓝色** | 主操作、选中、链接、信息 | #2563eb | #3b82f6 |
| **绿色** | 就绪、成功、正常、连接 | #059669 | #10b981 |
| **琥珀色** | 运行中、警告、需注意 | #d97706 | #f59e0b |
| **红色** | 错误、危险、紧急中止 | #dc2626 | #ef4444 |

每种语义色需要三个变体：
- **正常**：用于文字、图标、线条
- **淡底**：用于背景（状态徽章底色、输入框验证底色）
- **边框**：用于有底色元素的边框

```
以绿色为例：
  浅色主题: 正常 #059669 / 淡底 #ecfdf5 / 边框 #a7f3d0
  深色主题: 正常 #10b981 / 淡底 rgba(16,185,129,0.15) / 边框 rgba(16,185,129,0.25)
```

#### 背景层次

用2-3级背景色创造深度感，区分不同层级的容器：

```
浅色主题：
  页面底色  #f8f9fb  (最深的灰白)
  卡片/面板  #ffffff  (纯白，从底色中浮起)
  输入框    #f5f6f8  (略深于白，内嵌感)

深色主题：
  页面底色  #0a0e17  (最深)
  卡片/面板  #1a2234  (稍浅，浮起)
  输入框    #0d1321  (比卡片深，内嵌感)
```

不要用纯黑(#000)或纯白(#fff)作为大面积背景——纯黑/纯白对比度过高，长时间看会视觉疲劳。

#### 文字层次

三级文字色就够了，不要更多：

| 级别 | 用途 | 浅色主题 | 深色主题 |
|------|------|---------|---------|
| 主要 | 标题、关键数值 | #1a1d25 | #e2e8f0 |
| 次要 | 标签、说明文字 | #5a6175 | #8b9fc5 |
| 弱化 | 单位、占位符、次要信息 | #8c91a3 | #4a5f85 |

#### 图表配色策略

图表的配色取决于数据的语义。三种常见模式：

**时间演化（同一变量随时间/周期变化）：** 用单色调从浅到深（旧数据透明→新数据鲜明），或者用色相渐变（蓝→红表示从早到晚）。让用户一眼看出"时间方向"。

**分类对比（不同通道/信号/条件）：** 用语义上有区分的色相——蓝/红/绿/橙等。避免使用太接近的颜色（如蓝和青），色盲用户无法区分。同时配合线型（实线/虚线/点线）做冗余编码。

**极性/方向编码：** 正负极性、升降方向等成对概念用蓝/红是最直觉的映射（物理学惯例）。配合线型（实线/虚线）做冗余区分。

图表的中性元素（网格线、刻度、轴标题）应该非常克制：
- 网格线用最淡的灰（浅色主题 #f0f1f4，深色 #1e2d45），比边框还淡
- 刻度文字用弱化色
- 不要给图表加重边框或阴影——图表内容本身就是视觉焦点

#### 不要做的事

- 不要用渐变色做背景（除非按钮hover效果）
- 不要用饱和度过高的大面积色块
- 不要用超过4种语义色（蓝/绿/琥珀/红足够所有场景）
- 不要用颜色作为唯一信息编码——始终配合文字/图标/线型做冗余（考虑色盲用户和黑白打印）
- 不要用红色做常规操作按钮——红色保留给"危险/不可逆"操作（如Abort）
- 不要让Abort按钮在默认状态就很醒目——它应该低调地存在，hover时才变红

### 2.4 字体

#### 双字体系统

仪器软件需要且只需要两套字体：

**等宽字体——用于所有数值和数据。** 推荐：IBM Plex Mono、JetBrains Mono、SF Mono、Fira Code。

使用场景：参数输入框、实时数值显示、图表刻度标签、事件日志、仪器地址/型号、任何需要对齐的数字。

等宽字体的关键优势：数字宽度固定，数值变化时不会导致布局跳动。比如"1.23"和"9.99"宽度完全一样。

**无衬线字体——用于所有UI文字。** 推荐：Outfit、DM Sans、Inter、Plus Jakarta Sans。

使用场景：按钮文字、卡片标题、标签、描述文字、导航。

#### 字号层次

不需要太多层级，4-5级就够：

| 角色 | 字号 | 字体 | 场景 |
|------|------|------|------|
| 大数值 | 18-22px, Bold | 等宽 | 实时状态卡片里的关键数字 |
| 正文数值 | 12-14px, Regular/Medium | 等宽 | 输入框、图表刻度、日志 |
| UI标题 | 13-15px, SemiBold | 无衬线 | 卡片标题、图表标题 |
| UI正文 | 12-13px, Regular | 无衬线 | 参数标签、按钮文字 |
| 辅助文字 | 10-11px, Medium | 无衬线或等宽 | 单位、section标题(大写)、副标题 |

#### 数值显示的稳定性

数值在快速更新时不能导致UI抖动。做法：

- 数值区域预留固定宽度（根据最大可能的数字长度设定`min-width`）
- 使用等宽字体 + 右对齐（或小数点对齐）
- 单位跟在数字后面但视觉上弱化（小字号、浅颜色）
- 数值跳动时不要做动画/过渡——测量数据是实时的，动画反而让人觉得数据不准

### 2.5 减少参数输入

仪器软件普遍要求输入大量参数。每多一个需要手动输入的参数，就多一个出错的机会。

**Preset系统。** 为常见实验配置提供预设：选择preset后所有参数自动填入合理默认值。支持保存自定义preset（导出JSON）和导入。Preset选择器放在参数面板最顶部最醒目的位置。用户修改了参数后，在preset名称旁标记"(modified)"。

**参数分级。** 将参数分为"常用"和"高级"两级。常用参数（每次实验最可能调整的3-5个）始终可见；高级参数（有合理默认值、不常修改的）默认折叠在"Advanced"区域。判断标准：如果一个参数99%的情况下用默认值就行，它就是高级参数。

**智能默认值。** 每个参数都应该有一个合理的默认值，不能留空。默认值的来源优先级：Preset > 上次实验的值 > 文献参考值 > 工程经验值。

**参数联动。** 某些参数之间有关联：修改了其中一个，另一个的推荐范围可能变化。例如切换样品类型后，电流限值的合理范围完全不同。软件应能提示这种关联。

### 2.6 防止参数错误

**三级实时验证。**

| 级别 | 视觉 | 效果 | 典型场景 |
|------|------|------|---------|
| ❌ 错误 | 红色边框 + 红色提示文字 | 阻止启动 | 值超仪器物理极限、值为负、逻辑矛盾（step > max） |
| ⚠️ 警告 | 琥珀色边框 + 琥珀色提示 | 允许启动但弹确认 | 值异常大/小、可能损坏样品、可能导致无效实验 |
| ℹ️ 提示 | 灰色小字/tooltip | 仅提供信息 | 参数含义、典型范围、文献参考值 |

验证应在输入时**实时**执行（不是点Start才检查），用户能立刻看到哪个参数有问题。

**派生信息实时显示。** 用户填完参数后自动计算并显示有用的派生值，帮助用户快速判断参数是否合理：

常见的派生值：
- 单次测量/扫描的步数
- 估计耗时
- 最大功率/电流密度
- 与文献典型值的对比

这些值随参数修改即时更新，无需用户心算。

### 2.7 启动确认面板（Pre-flight Check）

点击Start后不应直接开始实验。弹出一个确认面板，整合三项功能：

**（1）参数总览。** 列出5-6个关键参数。与preset默认值不同的项高亮标注，让用户一眼看出"我改了什么"。

**（2）连通性自检。** 自动执行一次快速检测（施加微小激励、读取响应），在面板上显示结果：
- ✅ 仪器已连接，型号确认
- ✅ 样品检测到，初始电阻 = XXX Ω
- ❌ 开路，请检查探针/引线连接
- ⚠️ 电阻异常低（可能短路）

如果自检失败，Confirm按钮禁用。

**（3）实验元数据。** 必填字段：样品编号/ID、简短描述。可选字段：操作者、环境条件备注。不填不让开始。元数据随数据文件一起保存，保证三个月后还能知道这组数据是什么。

只有三项全部通过，用户才能点Confirm真正开始实验。

### 2.8 数值显示格式

仪器数据的动态范围极大（电流：A → pA 跨12个量级）。数值显示必须自适应。

**永远使用工程记数（SI前缀），永远不显示科学记数法（e记法）。** 4.27 kΩ 而不是 4270 Ω 或 4.27e3 Ω。1.20 nA 而不是 1.2e-9 A。

格式化规则：
- 选择使数字落在 1.00–999 范围内的SI前缀（p n μ m · k M G）
- 保持3–4位有效数字，不多不少
- 等宽字体、右对齐，数字变化时不导致布局抖动
- 数值和单位之间有半角空格
- 比率/比例值显示为百分比（5.0% 而不是 0.05）

输入框应接受多种格式：
- 纯数字："0.1"
- 带SI前缀："100m"（= 0.1）
- 科学记数法："1e-1"（= 0.1）
- 百分比（比率参数）："5%"（= 0.05）

输入框失焦后自动标准化为统一格式。

**图表轴刻度也应自适应。** Y轴单位随数据范围切换（Ω → kΩ → MΩ），轴标题随之更新，刻度数字保持简洁。

### 2.9 实验运行中的注意事项

**参数保护。** 运行中将参数分为两类：
- 不可修改（修改需要重启实验）：激励步进、通道、接线方式等
- 可运行时修改：阈值、终止条件等判据参数

不可修改的参数输入框变为只读灰色。可修改的保持可编辑，修改后即时生效并记录到操作日志。

**用户离开时的保障。** 实验可能持续数分钟到数小时，用户不会一直盯着。

- 关键事件声音提醒（compliance hit、实验完成、异常中止），一个简短提示音即可
- 浏览器/窗口标题动态更新状态（如"[Cycle 42] iFBEM"或"⚠️ Completed"）
- 实验运行中关闭UI弹`beforeunload`确认
- UI断连不应停止实验，后端独立运行，重连后恢复状态
- 关键阶段视觉升级（如接近目标时状态指示变得更醒目）

**操作日志。** 除了实验数据外，记录所有用户操作：什么时候改了什么参数、什么时候按了pause/resume、什么时候中止。存入事件日志文件，出问题可追溯。

### 2.10 数据安全

**永远不丢数据。** 这是仪器软件的底线。

- 每个数据点测量后立刻写入文件并flush，不攒buffer。程序崩溃最多丢最后一个点。
- 文件名包含时间戳，永远不覆盖已有文件。文件名冲突时自动加后缀。
- 每次实验的完整参数配置（含preset名称、所有参数值、元数据）自动保存一份JSON，与数据文件放在同一目录，保证可复现。
- 数据文件建议分为三个：原始数据（每个测量点）、周期总结（每个扫描周期）、事件日志（阈值触发、compliance、用户操作）。

**数据与参数绑定。** 数据文件头部或配套JSON包含完整的实验参数。后续分析时自动读取参数，不让用户手动输入。避免"拿错参数去分析"的错误。

### 2.11 崩溃恢复

- 程序启动时检测仪器输出状态，如果输出开着（上次崩溃遗留），先安全关闭并告警
- 通信中断自动重连，重连失败N次后报错并安全关闭仪器
- 定期存checkpoint（当前cycle、基线值、状态），恢复后可选继续或重新开始

## 5. 常见坑与调试经验

开发仪器控制软件时反复遇到的实际bug，按类别整理。

### 5.1 特殊浮点值（inf / nan / -0.0）

仪器测量经常产生特殊浮点值：开路时电阻 = `float('inf')`（电压 ÷ 零电流）、短路时可能出现 `nan`、负零 `-0.0` 等。这些值在Python内部运算没问题，但在序列化和传输时会引发一连串问题。

**JSON序列化。** `json.dumps(float('inf'))` 在默认设置下会输出 `Infinity`——这不是合法的JSON，前端 `JSON.parse()` 会直接报错。`float('nan')` 同理输出 `NaN`。

```python
# ❌ 会炸
json.dumps({"R": float('inf')})  # 输出 '{"R": Infinity}' — 非标准JSON

# ✅ 方案1：序列化前替换
def sanitize_for_json(value):
    """将inf/nan替换为None或字符串，安全序列化"""
    if isinstance(value, float):
        if math.isinf(value):
            return None  # 或 "Infinity" 作为字符串
        if math.isnan(value):
            return None
    return value

# ✅ 方案2：自定义JSON encoder
class SafeJSONEncoder(json.JSONEncoder):
    def default(self, obj):
        if isinstance(obj, float):
            if math.isinf(obj) or math.isnan(obj):
                return None
        return super().default(obj)

# ✅ 方案3：整个dict清洗
def sanitize_dict(d):
    return {k: sanitize_for_json(v) if isinstance(v, (int, float)) else v 
            for k, v in d.items()}
```

**WebSocket推送。** 如果用FastAPI的WebSocket发JSON，底层通常调用 `json.dumps`。必须在推送前清洗数据，否则前端收到非法JSON会断连。

**CSV写入。** Python的 `csv.writer` 会把 `float('inf')` 写成字符串 `"inf"`。读回来的时候 `float("inf")` 能正确解析回 `inf`，所以CSV层面问题不大，但要注意后续用Excel或其他工具打开时可能不识别。

**根本解决方案。** 在数据产生的源头就处理：

```python
# 在计算电阻时就防御
I, V = smu.measure_iv()
if abs(I) < config.noise_floor_current:
    R = float('inf')  # 明确标记为开路
    # 同时在数据dict里加一个标志字段
    data["open_circuit"] = True
else:
    R = abs(V / I)
    data["open_circuit"] = False
```

然后在序列化层统一处理 `inf`/`nan`，而不是在每个用到数据的地方各自处理。

### 5.2 仪器返回值解析

仪器返回的字符串格式经常不符合预期：

```python
# Keithley 2636B 的 measure.iv() 返回用制表符分隔
result = inst.query("print(smua.measure.iv())")
# 预期: "1.234e-03\t5.000e-01"
# 实际可能: "1.234e-03\t5.000e-01\n"  (尾部换行)
# 或者: "1.234e-03  5.000e-01"  (多个空格)
# 或者: "+1.234000E-03\t+5.000000E-01"  (不同精度格式)

# ❌ 脆弱的解析
I, V = result.split('\t')  # 如果不是tab分隔就炸

# ✅ 健壮的解析
parts = result.strip().split()  # 按任意空白符分割，去掉首尾空白
I, V = float(parts[0]), float(parts[1])
```

**布尔值返回。** 查询compliance状态时，不同仪器返回格式不同：

```python
# 可能返回 "true"、"false"、"1"、"0"、"1.0"、"0.0"
result = inst.query("print(smua.source.compliance)")

# ❌ 
is_compliance = result == "true"  # 如果返回 "1.0" 就漏了

# ✅ 
is_compliance = result.strip().lower() in ("true", "1", "1.0")
```

### 5.3 线程间数据竞争

测量线程和UI线程（或WebSocket handler）共享数据时：

```python
# ❌ 直接共享可变对象
class Experiment:
    def __init__(self):
        self.current_data = {}  # 测量线程写，UI线程读
    
    def measure_loop(self):
        self.current_data["R"] = new_R  # 写
    
    def get_status(self):
        return self.current_data  # 读 — 可能读到写了一半的状态

# ✅ 用Queue传递不可变数据的拷贝
data_queue.put({"R": new_R, "V": V, "I": I, "t": t})  # 每次创建新dict
```

`queue.Queue` 是线程安全的，但注意不要在queue里传递可变对象的引用——传进去之后如果还在修改原对象，接收端拿到的也会变。传dict的浅拷贝或者用dataclass/namedtuple。

### 5.4 VISA通信时序

```python
# ❌ 连续发送不等待
inst.write("smua.source.levelv = 0.5")
inst.write("smua.source.output = smua.OUTPUT_ON")
result = inst.query("print(smua.measure.iv())")
# 仪器可能还没设好电压就开始测量了

# ✅ 关键操作后加短延迟或用OPC
inst.write("smua.source.levelv = 0.5")
inst.write("smua.source.output = smua.OUTPUT_ON")
time.sleep(0.01)  # 给仪器一点settling time
inst.write("waitcomplete()")  # TSP的等待命令
result = inst.query("print(smua.measure.iv())")
```

另外，VISA超时设置不当是最常见的通信错误来源。`inst.timeout` 太短会在仪器正常执行长操作时误报超时；太长会在真正出问题时卡很久。建议默认5秒，长操作（如大缓冲区读取）临时调大。

### 5.5 文件句柄泄漏

```python
# ❌ 实验中途崩溃，CSV文件没关
csv_file = open("data.csv", "w")
writer = csv.writer(csv_file)
# ... 实验运行 ...
# 崩溃了 → csv_file 没关 → 数据可能丢失

# ✅ 方案1：每次写完flush
writer.writerow(data)
csv_file.flush()  # 确保写入磁盘

# ✅ 方案2：用atexit注册清理
import atexit
atexit.register(csv_file.close)

# ✅ 方案3：在DataStorage类的__del__里关闭
```

### 5.6 前端数值精度

JavaScript的Number类型是IEEE 754双精度浮点，大多数情况足够。但要注意：

```javascript
// ❌ 浮点比较
if (resistance === 4.27) { ... }  // 几乎永远为false

// ✅ 
if (Math.abs(resistance - 4.27) < 1e-10) { ... }

// ❌ 显示时直接toString
element.textContent = resistance;  // 可能显示 4.269999999999999

// ✅ 用格式化函数
element.textContent = formatValue(resistance, "Ω");
```

### 5.7 Chart.js 性能

实时图表数据量大时Chart.js会变慢：

```javascript
// ❌ 每个数据点都调用chart.update()
ws.onmessage = (msg) => {
    chart.data.datasets[0].data.push(newPoint);
    chart.update();  // 每个点都重绘 → 卡顿
};

// ✅ 攒一批再更新，或用requestAnimationFrame
let pendingPoints = [];
ws.onmessage = (msg) => {
    pendingPoints.push(newPoint);
};

function renderLoop() {
    if (pendingPoints.length > 0) {
        chart.data.datasets[0].data.push(...pendingPoints);
        pendingPoints = [];
        chart.update('none');  // 'none'跳过动画，更快
    }
    requestAnimationFrame(renderLoop);
}
requestAnimationFrame(renderLoop);

// ✅ 限制数据点数量
const MAX_POINTS = 2000;
if (chart.data.datasets[0].data.length > MAX_POINTS) {
    chart.data.datasets[0].data.splice(0, chart.data.datasets[0].data.length - MAX_POINTS);
}
```
